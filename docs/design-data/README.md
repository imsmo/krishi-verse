# Phase-1 Design Data (all 196 screens)

This folder is the **single source of truth for the demo/seed data shown in the Phase-1 designs**
(`Phase-1 all screen design/Krishi_Verse_Design_System/screens/*.html`). Use it to build every screen
"same to same" as the design, with the exact names, prices, crops, villages and copy the designer used.

## Files

| File | What it is | Use it for |
|------|------------|-----------|
| `SCREEN-DATA-CATALOG.json` | Machine-readable extraction of **every visible string** from all 196 screens, keyed by screen id. | Look up exactly what a given screen renders (e.g. `09-farmer-home`). |
| `SCREEN-DATA-CATALOG.md` | The same data, human-readable, grouped by role/module. | Browsing / reference while building a screen. |
| `demo-dataset.json` | **Curated, entity-organized** demo data (users, listings, mandi prices, orders, wallet, transactions, auctions, workers, bookings, schemes, tips, notifications, ambassador) with **stable demo UUIDs** and money in **both** a display string and `*_minor` paise (Law 2). | The actual seed: feed it into SQL seeds or mobile fixtures so screens render design-faithful data. |

## How to use this for seeding

1. **SQL seed (Postgres):** map a `demo-dataset.json` entity to its table using `db/migrations` columns, following the
   proven pattern in `db/local/local-demo-listings.sql` (fixed UUIDs, `ON CONFLICT DO NOTHING`, code-resolved refs).
   Note: `mandi_prices` / `price_predictions` are **partitioned by date** and FK to `products` ÔÇö seed those carefully
   (ensure a partition for the `price_date` exists and the crop's `product_id` is present from `db/seeds/catalogue/0103`).
2. **Mobile / web fixtures:** import the JSON directly as the offline/demo data layer while a screen is built against the
   design, then swap to the live SDK call once the API returns the same shape.
